## HIGH — 2명 이상이 지적

### HIGH-1 — 계정 전환 / 다운그레이드 시 stale cache
**지적**: senior H2 · codex H2 · quality M7 (3명 합의)

**현상**: `UsageViewModel`이 처음 enterprise라고 판단되면 `cachedTeamId`에 팀 ID를 저장하고, 그 뒤로는 `/api/dashboard/teams`를 다시 호출하지 않음. `weeklyData`, `isEnterpriseTeam`, jump 비교용 baseline (`previousPlanUsedCents` 등)도 일단 채워지면 logout 전까지 유지됨.

**왜 문제**: 사용자가 enterprise 팀에서 빠지거나 plan 다운그레이드를 하면, 캐시는 옛 팀 ID를 들고 있고 매 refresh마다 analytics endpoint에 403 응답이 올 텐데 코드의 catch가 그걸 조용히 삼킴. 결과적으로 차트는 마지막 성공한 데이터에 영구 freeze되고, Settings의 "Weekly Chart" 섹션도 계속 보임.

**시나리오**: 팀 admin이 사용자를 팀에서 제거한 다음 날 앱을 켰을 때 → 옛 팀의 stale 데이터 그대로 표시 + 새 plan에 맞지 않는 UI 노출.

**수정 방향**: (a) analytics 호출이 403/404 반환하면 `cachedTeamId = nil`로 클리어하고 `isEnterpriseTeam = false` 설정 — codex가 제안한 가장 surgical fix. (b) `onLoginSuccess`에서도 weeklyData/jump baseline 다 리셋 (현재는 logout 시만 리셋).

### HIGH-2 — observeViewModel over-tracking → status item + popover 동시 redraw
**지적**: efficiency H1 · quality H3 (2명)

**현상**: `CursorMeterApp.observeViewModel()` 한 블럭이 10개 property를 동시에 track. 그 중 어느 하나라도 바뀌면 `updateStatusItem()` + `popover.updateUI()` 둘 다 호출됨.

**왜 문제**: 메뉴바 아이콘은 `usageData.percentUsed`와 `menuBarDisplayMode`에만 종속. 그런데 weeklyData/weeklyChartStyle 변경, refreshInterval 변경, errorMessage 토글 모두 NSImage를 새로 만들어서 status item.button.image에 set. AppKit이 같은 픽셀이라도 setter 호출에 redraw 트리거.

**시나리오**: 1분 refresh, enterprise 계정. weekly fetch가 매번 새 DayUsage 배열(equality 비교가 의미 없는 reference)을 만들어 `weeklyData`에 대입. observation tracking이 fire → 매번 menu bar pie chart NSImage 재생성.

**수정 방향**: tracking 블럭을 두 개로 분리: ① status-item-only (usageData, menuBarDisplayMode, authState, errorMessage, isLoading 일부, availableUpdate); ② popover-only (모든 weekly* + refreshInterval). 두 블럭이 각자 re-arm.

### HIGH-3 — 퍼센트→색 threshold ladder 3종 불일치
**지적**: reuse H1 · quality M11 (2명)

**현상**: 동일하게 "퍼센트 → 색"인데 세 곳이 다른 기준 사용:

| 위치 | 임계값 | 색상 출처 |
|---|---|---|
| `CircularProgressIcon.pieColor` (메뉴바 ring) | ≥90 red, ≥70 yellow | RGB 직접 (`accentColor`/`warnColor`/`critColor`) |
| `MenuBarView.progressNSColor` (popover progress bar) | ≥90 red, ≥80 yellow | system colors (`.systemRed` 등) |
| `WeeklyUsageChartView.heatColor` (주간 차트 막대) | ≥85 red, ≥60 yellow (weeklyMax 기준) | RGB tokens 재사용 |

**왜 문제**: 사용자가 같은 popover 안에서 "ring=초록 / progress bar=노랑"인 모순된 상태를 볼 수 있음 (예: 85%일 때 ring threshold=70 안 넘었으니 노랑 안 보일 수도 있고, progress bar는 80 넘었으니 노랑). 이슈 #60 spec의 "color tokens 공유 (drift 방지)" 의도와 그 안에서도 이미 drift 중.

**시나리오**: 사용량 80~89% 구간일 때 menu bar pie=노랑, popover bar=노랑, weekly chart의 가장 큰 막대=빨강 (weeklyMax의 85%니까). 세 가지 시각 신호가 동일 데이터에 대해 서로 다른 강도.

**수정 방향**: weekly chart는 weeklyMax 상대 기준이라 단순 통합 불가. 하지만 ring과 progress bar는 같은 plan-% 기준이니 통합 가능. `CircularProgressIcon.level(for:)`를 ring/progress bar 모두 호출하도록 하고 threshold도 한 곳 (70/90)으로 일치.

### HIGH-4 — UserDefaults keys 11개 stringly-typed + 중복
**지적**: reuse M3 · quality H1 (2명)

**현상**: `UsageViewModel`에 setter 11개. 각 setter는 `UserDefaults.standard.set(value, forKey: "literal-key")` 호출하고, 같은 키 literal이 `loadSettings()`에서도 `defaults.object(forKey: "literal-key")`로 다시 등장. 즉 키 11개 × 2번 = 22번의 string literal이 코드에 흩어져 있음.

**왜 문제**: setter에서 키를 typo로 잘못 적으면 save는 성공하는데 load 시 안 읽힘. 컴파일러는 잡지 못하고 테스트도 없음.

**시나리오**: 새 설정 추가하다가 `setWeeklyChartStyle`는 키 `"weeklyChartStyle"`로 save, 그런데 `loadSettings`에 `"weeklyChartstyle"` (소문자 s)로 read. 앱 재실행 시 사용자 선택이 항상 default로 돌아감 — 발견까지 며칠.

**수정 방향**: `private enum SettingsKey: String { case refreshInterval = "refreshIntervalSeconds", weeklyChartEnabled, ... }` 도입. 22번 literal → 11번 enum case 1번 정의로 압축.

## HIGH — 1명이 high confidence로 지적 (cross-comment 단계에서 fact-check 필요)

### HIGH-5 — NotificationManager 거부 상태 무한 ping loop
**지적**: senior H1

**현상**: `NotificationManager.sendNotification(threshold:level:)`은 매 호출마다 `UNUserNotificationCenter.requestAuthorization`를 부름. 사용자가 권한을 거부했으면 system이 즉시 `granted=false`로 응답하고 함수는 early return. 그런데 `notifiedThresholds.insert(threshold)`는 granted=true 분기에서만 실행되는 것으로 추정.

**시나리오**: 거부 상태에서 90%↑ 상태로 며칠 사용. 1분 refresh면 시간당 60회, 24시간이면 1440회의 권한 요청 시도.

**수정 방향**: (a) `notifyUsageJump`처럼 먼저 `authorizationStatus` 확인하고 `.denied`면 early return. (b) granted/denied 관계없이 `notifiedThresholds.insert(threshold)`는 통과시키기.

### HIGH-6 — NotificationManager 청구 사이클 롤오버 시 알림 영구 침묵
**지적**: codex H4

**현상**: `notifiedThresholds`가 한 번 80/90 받으면 앱이 다시 시작될 때까지 영구 보존. `resetNotifications()`는 logout 시에만 호출.

**왜 문제**: 매월 1일 사이클이 reset되면 percent도 0%로 떨어졌다가 다시 올라갈 텐데, 새 cycle에서 80% 다시 도달해도 "이미 알림 보냄"으로 판단해서 침묵.

**시나리오**: 월간 cycle 사용자. 1월에 90% 도달 → 빨강 notification. 2월 1일 reset → 2월 중 다시 90% 도달 → 알림 없음.

**수정 방향**: `UsageViewModel.refresh()`에서 `cycleStartDate` 변경을 감지해 `notificationManager.resetNotifications()` 호출.

### HIGH-7 — 오프라인 retry 분기 누락 (APIError.networkError 언래핑)
**지적**: codex H1

**현상**: `CursorAPIClient.performRequest`는 `URLSession`이 throw한 `URLError`를 `APIError.networkError(URLError)`로 wrap해서 다시 throw. `UsageViewModel.refresh()`의 catch 분기는 raw `URLError`만 검사:

```swift
let urlError = error as? URLError  // ← .networkError 안에 든 URLError는 nil
if urlError?.code == .notConnectedToInternet || ... {
    errorMessage = "Waiting for network..."
    scheduleNetworkRetry()    // ← 도달 못함
}
```

**시나리오**: 노트북 닫고 카페에서 다시 열 때. Wi-Fi 재연결 직후 1분간은 자동 retry되어야 하는데, 사용자가 popover 열고 refresh 버튼 누르기 전까지 stale 상태.

**수정 방향**: `refresh()`의 catch 분기에서 `case APIError.networkError(let underlying) = error` 추출해 `underlying as? URLError`도 같이 검사.

### HIGH-8 — Jump emoji swap timer 중 ring 조기 복원 race
**지적**: codex H3

**현상**: jump 발생 → `lastJump` 설정 → `JumpEffectCoordinator`가 status item.button.image를 ⚡/🚀 이모지로 swap + 6/15초 timer 시작. 그런데 같은 refresh에서 `weeklyData`, `isEnterpriseTeam`, `isLoading`도 같이 변경 → `observeViewModel` onChange → `updateStatusItem()` 호출 → ring 이미지로 다시 덮어씀.

**시나리오**: enterprise 계정에서 사용량 급증. 이모지 ⚡가 보였다 0.1초 만에 사라짐.

**수정 방향**: (a) `updateStatusItem()`이 swap 진행 중인지 확인하고 진행 중이면 ring 그리기 skip. (b) viewModel 변경 시 `updateStatusItem` 호출 순서를 `lastJump` 처리 뒤로 옮김.

### HIGH-9 — `lastJump = nil → nil` 매 refresh @Observable 발화
**지적**: efficiency H2

**현상**: `UsageViewModel.updateJumpState()`는 delta가 0 이하거나 첫 refresh일 때 `lastJump = nil` 대입. `@Observable`은 setter 호출 자체로 observer 발화 (값 dedup 안 함).

**왜 문제**: 불필요한 observer 발화. `JumpEffectCoordinator`가 매 refresh마다 잠을 깸.

**수정 방향**: `if lastJump != nil { lastJump = nil }` 가드 한 줄.

### HIGH-10 — enterprise weekly fetch sequential (parallel 가능)
**지적**: efficiency H3

**현상**: `UsageViewModel.refresh()`는 summary/usage/userInfo 3개를 `async let`으로 parallel fetch. 그 batch 완료 후 그 다음에 `refreshWeeklyChart` 호출 → 내부에서 (cachedTeamId 있으면) 또 fetchWeeklyUsage 하나 더.

**왜 문제**: cachedTeamId가 이미 있으면 fetchWeeklyUsage는 다른 3개와 동시 실행 가능. 현재는 시리얼이라 enterprise 계정 refresh가 한 round trip(200-600ms) 더 걸림.

**수정 방향**: cachedTeamId가 있을 때만 `async let weeklyResult = ...`를 primary batch와 병렬로 시작.

### HIGH-11 — UsageDisplayData factory 두 path drift
**지적**: quality H2

**현상**: `UsageDisplayData.from(summary:usage:userInfo:)`와 `from(usage:userInfo:)`가 60% 동일한 본문. resetDate 계산만 서로 다름.

**왜 문제**: 새 필드 추가 시 두 곳 모두 업데이트 필요. 한 군데 빠지면 어느 경로 사용자가 새 필드를 못 받음.

**수정 방향**: 공통 부분 (userInfo defaults, daysUntilReset 계산)을 헬퍼로 추출.

### HIGH-12 — `nonisolated(unsafe)` iso8601 enforcement gap
**지적**: senior H3

**현상**: `UsageDisplayData.iso8601`은 `nonisolated(unsafe) static let`으로 선언. `DateFormatter`/`ISO8601DateFormatter`는 thread-safe하지 않음. 현재 caller는 모두 `@MainActor`라 실제 race는 없지만, `unsafe` annotation이 컴파일러의 Swift 6 검사를 침묵시킴.

**수정 방향**: 두 정적 팩토리 메소드에 `@MainActor` 추가. `nonisolated(unsafe)` 제거.

---

### HIGH-13 — `yyyy-MM-dd` formatter 3곳 중복 생성
**지적**: reuse H2

**현상**: 동일 포맷 `yyyy-MM-dd`를 다음 3곳에서 각자 만듦:
- `WeeklyUsageChartView.weekdayFormatter` (static let)
- `UsageViewModel.weeklyDateFormatter` (static let)
- `WeeklyUsageResponse.sevenDayRolling`은 매 호출마다 새 인스턴스 생성

**수정 방향**: 하나의 `static let utcDayFormatter`를 `WeeklyUsageResponse` 안 또는 `DateFormatters` enum에 정의하고 세 곳이 공유.
